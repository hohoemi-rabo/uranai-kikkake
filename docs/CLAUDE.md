# docs/CLAUDE.md

`docs/` 配下のチケット運用ガイド。

## TODO 記法

各チケット内のチェックリストは:

- **未完了**: `- [ ]`
- **完了**: `- [×]`(`x` ではなく全角クロス `×`(U+00D7))

完了は項目単位でその場で書き換える。**チケット内のすべての TODO が `- [×]` になり、受入基準を満たしたらチケット完了。**

`00-INDEX.md` には進捗マークを付けない(各チケットを開けば分かる)。

## チケット構成テンプレート

新規チケット作成時はこのテンプレートに従う:

```markdown
# NN. <タイトル>

> 関連: REQUIREMENTS.md §X.X / 他チケット名

## 概要
<2〜3 行で目的を述べる>

## 前提
- [NN. <依存チケット名>](./NN-<file>.md)
- (なければ「なし」と書く)

## TODO
- [ ] ...
- [ ] ...

## 受入基準
- ...

## 技術メモ
- 注意点・参考情報・引っかかりやすいポイント
```

## 番号運用

- 番号は**チケット ID** であり、必ずしも着手順ではない
- 着手順の目安は `00-INDEX.md` に記載
- 新規追加は **既存最大番号+1** を使う(過去番号の再利用はしない)
- 廃止・延期チケットはファイル削除せず、冒頭に「ステータス」セクションを書いて理由を残す(`05-auth-apple-phase2.md` のフェーズ 2 延期パターンと同じ)

## 仕様変更時のルール

新しい仕様や追加要件が出たら、**対応する `docs/NN-*.md` の TODO に項目を追加**する。仕様だけ口頭で増やして TODO に反映しないと、後続の Claude が拾えなくなる。

仕様の出所(REQUIREMENTS のセクション、Slack スレッド、ユーザーリクエスト等)を「関連」に追記すること。

## 受入基準の書き方

- **機能観点**で書く(「ボタンを押すと X が起きる」)
- **計測可能**な内容にする(「○○が 1 秒以内に表示される」)
- **ユーザー体験ベース**で書き、内部実装に依存させない

## INDEX 更新

新規チケット追加・ステータス変更時は `00-INDEX.md` も更新する。INDEX の役割:

- 全チケットへのリンク一覧(セクション分け)
- 着手順の目安(矢印フロー)
- フェーズ 2 などのステータス区分

## 既存チケットのリファレンス用途

各チケットの「技術メモ」は、未来の Claude が同じ罠にハマらないようヒントを残す場所。既知の落とし穴・SDK 互換性・Web 版との差分などを書く。

## チケット完了時の CLAUDE.md 自発更新

チケットのすべての TODO が `- [×]` になり受入基準を満たしたら、**ユーザーに依頼される前に** 該当階層の `CLAUDE.md` に学びを反映する(ルート CLAUDE.md の「振り分けルール」表を参照)。

書くべきこと:
- ライブラリの非自明な挙動(SDK のバージョン互換性、ドキュメントに書いていない落とし穴)
- 試して失敗した方針と正解(同じ試行錯誤を繰り返さないため)
- 製品コードに残った暗黙の前提(なぜこの形になっているか)

書かないこと:
- チケット自体の内容(チケットを読めば分かること)
- ファイル構成や API 仕様(コードを読めば分かること)
- 一回限りの作業手順
